# AGENTS.md — Chess Trainer Project

This file provides context, architecture decisions, and coding guidelines for AI coding agents working on this project. Read this fully before making any changes.

---

## Project Overview

A chess opening training web application built for a small number of users (2–3 concurrent max). The goal is not to play chess games against others, but to train and test opening knowledge against an opening book database. Users authenticate, select or build opening repertoires (playbooks), and drill against the app which throws sidelines and tests depth using master-game data.

Planned features (in rough priority order):
1. Opening training against a user-defined or suggested opening book
2. Repertoire/playbook management (supporting lines for both colors)
3. Puzzle trainer (free, unlimited)
4. Position concept questions ("what are the main ideas here?") — future/experimental
5. Optional 3D board perspective mode — future

## Database Schema

The database is hosted on Supabase (Postgres). All tables have Row Level Security (RLS) enabled.

### Tables

**`profiles`** — extends `auth.users`, stores display info
```sql
id uuid (PK, FK → auth.users)
username text (unique)
created_at timestamptz
updated_at timestamptz
```

**`opening_books`** — user repertoires/playbooks
```sql
id uuid (PK)
user_id uuid (FK → profiles)
name text
color text ('white' | 'black')
moves jsonb          -- opening tree stored as nested JSON
is_public boolean
created_at timestamptz
updated_at timestamptz
```

**`training_sessions`** — results of each training drill
```sql
id uuid (PK)
user_id uuid (FK → profiles)
book_id uuid (FK → opening_books, nullable)
result text ('pass' | 'fail' | 'abandoned')
moves_played jsonb   -- array of moves made during session
correct_moves int
total_moves int
duration_seconds int
created_at timestamptz
```

**`position_cache`** — cached Lichess Opening Explorer API responses
```sql
fen text (PK)        -- FEN string is the unique position key
explorer_data jsonb  -- raw API response
cached_at timestamptz
```

**`puzzles`** — imported from Lichess puzzle database dump
```sql
id text (PK)         -- Lichess puzzle ID
fen text
moves text[]         -- solution in UCI format
rating int
themes text[]
popularity int
```

**`puzzle_history`** — per-user puzzle attempt tracking
```sql
id uuid (PK)
user_id uuid (FK → profiles)
puzzle_id text (FK → puzzles)
solved boolean
time_seconds int
attempted_at timestamptz
UNIQUE(user_id, puzzle_id)
```

### RLS Policies Summary
- `profiles`: users read/write their own row only
- `opening_books`: users manage their own; all authenticated users can read public books
- `training_sessions`: private to each user
- `puzzle_history`: private to each user
- `position_cache`: readable by all authenticated users, written only by server (service role)
- `puzzles`: readable by all authenticated users

## Schema Migration Workflow

**Never change the database schema by clicking in the Supabase UI without also recording it as a migration file.**

Migration files live in `supabase/migrations/` and are named with a timestamp prefix:
```
supabase/migrations/
├── 20260419_001_initial_schema.sql
├── 20260419_002_add_notes_to_session.sql
└── ...
```

### When making a schema change:
1. Write a new `.sql` file in `supabase/migrations/` describing only the change (not the full schema)
2. Run it in the Supabase SQL Editor
3. Regenerate TypeScript types:
   ```bash
   npx supabase gen types typescript --project-id alyrmmmaxbcxgbphjcag > src/types/database.ts
   ```
4. Commit both the migration file and the updated `src/types/database.ts`

### `src/types/database.ts` is auto-generated
Do not hand-edit this file. It is always regenerated from the live Supabase schema using the CLI command above. If types look wrong, re-run the command rather than editing the file.

### Prisma (future consideration)
Prisma may be introduced later as an ORM layer for type-safe queries and a more structured migration workflow. If/when Prisma is added:
- A `prisma/schema.prisma` file will define the data model
- `prisma migrate dev` replaces the manual SQL migration workflow
- `src/types/database.ts` may be replaced by Prisma's generated client types
- Do not mix Prisma and raw Supabase client queries in the same table's data access layer — pick one per table and be consistent

Until Prisma is introduced, all DB access goes through the Supabase JS client in `src/lib/db/`.

## Key Architectural Decisions

### Stockfish runs client-side, not server-side
The chess engine (Stockfish 18 WASM) runs entirely in the user's browser via a Web Worker. There is no server-side engine process. This means:
- Zero server compute cost for analysis
- No queue system needed — each user's browser runs its own engine instance
- The `src/workers/stockfish.worker.ts` file manages communication with the WASM binary
- The `useEngine` hook provides a clean interface to the worker from React components
- Vercel must serve the app with `Cross-Origin-Embedder-Policy: require-corp` and `Cross-Origin-Opener-Policy: same-origin` headers for the multi-threaded engine flavor to work. Configure this in `next.config.ts`.

### Lichess Opening Explorer is proxied, not called directly from the client
All calls to `explorer.lichess.ovh` go through `src/app/api/openings/explorer/route.ts`. This API route:
1. Checks `position_cache` in Supabase first (cache hit → return immediately)
2. On cache miss, calls the Lichess API
3. Stores the result in `position_cache` before returning
4. Respects Lichess rate limits (back off on 429 responses)

Never call the Lichess API directly from client components.

### Opening books are stored as JSONB trees
A user's opening repertoire is stored as a single JSONB column (`moves`) in `opening_books`, not as relational rows. The tree structure represents the move graph. This keeps queries simple and the data portable. A full GM-depth repertoire for one color is typically only a few hundred KB.

### Auth pattern
NextAuth v5 beta with Supabase as the database backend. Config lives in `src/lib/auth.ts`. Route protection is handled in `src/middleware.ts` (equivalent of `proxy.ts` in the reference project). All `/dashboard/*` routes require authentication.

### API route pattern
```
Client component
  → fetch('/api/openings/explorer', { method: 'POST', body: { fen } })
  → src/app/api/openings/explorer/route.ts
  → src/lib/db/positionCache.ts (cache check)
  → src/lib/chess/lichessExplorer.ts (if cache miss)
  → returns data to client
```

Keep business logic out of route handlers. Route handlers should validate input (Zod), call lib functions, and return responses. Logic lives in `src/lib/`.

## Coding Conventions

- **TypeScript strict mode** — no `any` types. Use types from `src/types/` and the auto-generated `database.ts`.
- **Zod for all API input validation** — schemas live in `src/lib/validators/schemas.ts`.
- **No direct DB calls from components** — components call API routes or hooks; hooks call API routes; API routes call `src/lib/db/` functions.
- **FEN strings are the universal position identifier** — use FEN as cache keys, lookup keys, and position references throughout. Never use move number + move text as a position identifier.
- **UCI format for engine moves** — Stockfish speaks UCI (e.g. `e2e4`). Convert to SAN (e.g. `e4`) for display using `chess.js` only at the UI layer.
- **Tailwind v4** for all styling — no CSS modules, no inline styles unless absolutely necessary.
- **Server Components by default** — only add `'use client'` when the component needs browser APIs, event handlers, or React hooks. The chessboard and engine components will always be client components.

## External APIs

### Lichess Opening Explorer
- Base URL: `https://explorer.lichess.ovh`
- Master games endpoint: `GET /master?fen=<FEN>&moves=<N>`
- Returns: move options with white/draw/black counts from master games
- Rate limit: back off on HTTP 429, wait 60 seconds before retrying
- Always proxy through `/api/openings/explorer` — never call from client

### Lichess Opening Names
- Static dataset from `https://github.com/lichess-org/chess-openings`
- Bundle as a JSON file in the project rather than fetching at runtime
- Used to label positions with ECO codes and opening names

### Lichess Puzzle Database
- Downloaded from `https://database.lichess.org/#puzzles`
- Imported as a CSV into the `puzzles` table
- No runtime API calls needed for puzzles — all data lives in Supabase

## What Not to Do

- Do not run Stockfish on the server. Engine analysis is client-side only.
- Do not call `explorer.lichess.ovh` directly from client components.
- Do not hand-edit `src/types/database.ts`.
- Do not change the Supabase schema without writing a migration file.
- Do not put `SUPABASE_SERVICE_ROLE_KEY` in any `NEXT_PUBLIC_` variable or reference it in client components.
- Do not store opening tree moves as relational rows — use the JSONB `moves` column in `opening_books`.
- Do not use the React Compiler — it is not enabled in this project.
